## Этап 1: Критические исправления (1-3 дня)

### 1. Исправление Sharpe Ratio
**Файлы:** `backtest/metrics.py:48-54`, `backtest/portfolio.py:238-242`

**Проблема:** Sharpe считается по доходностям на сделку, а аннуализуется через `sqrt(252)` (как для дневных). Завышает/занижает Sharpe в зависимости от частоты сделок.

**Решение:**
- Построить дневной equity curve из сделок (группировать по `exit_date`)
- Считать daily returns, затем применять `sqrt(252)`
- Использовать `ddof=1` (sample std)
- Обновить тесты в `tests/test_metrics.py`

### 2. Max Drawdown по календарю
**Файлы:** `backtest/metrics.py:35-46`, `backtest/portfolio.py:226-235`

**Проблема:** Equity curve строится в порядке закрытия сделок, а не по календарю. Период без сделок не показывает просадку.

**Решение:**
- Строить equity curve по календарным барам
- Учитывать mark-to-market для открытых позиций
- Обновить тесты

### 3. Подключение продвинутых метрик
**Файлы:** `backtest/engine.py:13`, `analytics/performance.py`

**Проблема:** `analytics/performance.py` содержит Sortino, Calmar, VaR, CVaR, Ulcer Index, UPI, Kelly — но движок импортирует только базовый `backtest/metrics.py`.

**Решение:**
- Заменить импорт в `backtest/engine.py` на `analytics.performance`
- Проверить совместимость возвращаемых ключей с UI (`visual.py`)
- Добавить Sortino, Calmar, VaR, CVaR в отображение

### 4. Исправление Kelly sizing
**Файлы:** `backtest/engine.py:80-93`

**Проблема:** Kelly формула применяется как доля капитала для аллокации, а не как доля риска. При тугом SL недориск, при широком — перериск.

**Решение:**
- Заменить формулу на `qty = (capital * f_kelly * kelly_fraction) / abs(entry_price - sl_price)`
- Добавить тест на корректность размера позиции

### 5. Добавление slippage (зависимость от ликвидности)
**Файлы:** `backtest/engine.py`, `intraday/engine.py`, `strategy/config.py`

**Проблема:** Нет моделирования проскальзывания. Только комиссия. Для MOEX реальное проскальзывание 5-20 bps на сторону.

**Решение:**
- Новый параметр `slippage_bps` (базовое значение по умолчанию: 5)
- Рассчитывать фактический slippage в зависимости от ликвидности тикера:
  - Измерять средний дневной объём за последние N свечей
  - Чем ниже ликвидность — тем выше slippage (линейная или ступенчатая шкала)
  - Формула: `slippage = base_bps * max(1, liquidity_factor)` где `liquidity_factor` обратно пропорционален среднему объёму
- Применять к цене входа и выхода: `entry *= (1+slip)` для BUY, `(1-slip)` для SELL
- Добавить параметр в UI и `strategy/config.py`
- Добавить тесты

### 6. SL/TP intra-bar неоднозначность
**Файлы:** `backtest/engine.py:358-371`, `intraday/engine.py:146-159`

**Проблема:** Когда SL и TP попадают в диапазон `[low, high]` одной свечи, движок всегда считает, что SL сработал первым.

**Решение:**
- Документировать текущий консервативный подход
- Добавить параметр `exit_assumption` (conservative/optimistic/random)
- В консервативном режиме: SL первым для BUY, TP первым для SELL
- В optimistic режиме: наоборот
- В random режиме: 50/50 с усреднением

## Этап 2: Реалистичность (2-3 дня)

### 7. Survivorship bias — документирование
**Файлы:** `core/moex_cache.py`, `intraday/smart_scanner.py`, `README.md`

**Проблема:** Тестируем только текущие голубые фишки TQBR — исключены делистинговые акции. Завышает доходность.

**Решение (этап документирования):**
- Документировать ограничение TQBR в README и `docs/ARCHITECTURE.md`
- Добавить предупреждение в UI при запуске сканера: "Внимание: используется текущий состав TQBR, возможен survivorship bias"
- Добавить примечание в результаты скрининга

**Будущее (отдельная задача):**
- Добавить загрузку исторического состава индекса MOEX
- Использовать point-in-time состав борда

### 8. Walk-forward score — принципиальная метрика
**Файлы:** `optimization/grid.py:137-152`

**Проблема:** PF входит в формулу дважды. Нет статистического порога. Минимум 5 сделок — слишком мало.

**Решение:**
- Убрать дублирование PF
- Новая формула: `score = sharpe * 0.4 + log(pf + 1) * 0.3 - max_dd * 0.3`
- Минимальный порог сделок: 30 (вместо 5)
- Обновить тесты

### 9. OOS валидация в grid search (фиксированный 80/20)
**Файлы:** `optimization/grid.py:155-202`

**Проблема:** `optimize()` возвращает топ-50 по in-sample без train/test split.

**Решение:**
- Разделить данные на фиксированные 80% IS / 20% OOS
- Запускать оптимизацию на IS (первые 80%)
- Прогонять топ-50 на OOS (последние 20%)
- Возвращать оба набора метрик
- Флагировать результаты с деградацией OOS > 50% vs IS
- Добавить визуализацию IS vs OOS в UI

### 10. Sensitivity heatmap — реализация
**Файлы:** `optimization/sensitivity.py:212-241`

**Проблема:** `plot_heatmap` всегда возвращает `None`.

**Решение:**
- Реализовать 2D-сетку: итерировать `x_vals x y_vals`
- Запускать `BacktestEngine` для каждой комбинации
- Построить матрицу и `ax.imshow()`
- Добавить цветовую шкалу и аннотации

### 11. ATR sizing — опция risk_based
**Файлы:** `backtest/engine.py:74-78`

**Проблема:** Размер позиции считается по ATR, а не по дистанции до SL.

**Решение:**
- Добавить опцию `risk_mode`: `volatility` (текущий) или `risk_based`
- В режиме `risk_based`: `qty = risk_amount / abs(entry - sl)`
- Документировать разницу
- Добавить тесты

### 12. Лимит ликвидности на размер позиции
**Файлы:** `backtest/engine.py:74-100`, `intraday/engine.py:124-125`

**Проблема:** Ничто не ограничивает qty долей дневного объёма. Нереалистичные заполнения.

**Решение:**
- Вычислять средний дневной объём из последних N свечей
- Ограничивать `qty` на уровне 5% от среднего объёма
- Логировать отклонённые сделки
- Добавить параметр `max_volume_pct` (по умолчанию 5%)

## Этап 3: Качество стратегий (2-3 дня)

### 13. Bounce strategy — усиление логики уровня
**Файлы:** `strategy/bounce.py:18-35`

**Проблема:** Условие `low <= level + threshold` всегда выполняется при `low <= level`. Нет требования фактического касания уровня.

**Решение:**
- Требовать `low <= level` для BUY bounce (фактическое касание)
- Усилить проверку `prev_close` близости к уровню (например, в пределах 2 ATR)
- Обновить `tests/test_bounce.py`

### 14. Ковариация позиций в портфеле (Risk Parity)
**Файлы:** `backtest/portfolio.py`, `backtest/portfolio_risk.py`

**Проблема:** Портфельный бэктест делит капитал поровну и не учитывает корреляции между позициями.

**Решение:**
- Рассчитывать матрицу корреляций доходностей между тикерами
- Добавить опцию Risk Parity для распределения капитала (веса обратно пропорциональны волатильности)
- Добавить опцию Min-Variance (веса минимизируют портфельную дисперсию)
- Выводить портфельную Beta и корреляционную матрицу в UI
- Учитывать ковариацию при работе `PortfolioRiskManager`
- Добавить тесты
