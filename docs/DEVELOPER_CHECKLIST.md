# Developer Checklist

## Перед началом задачи

- понять, какой модуль меняется
- найти существующие тесты рядом с этой логикой
- проверить, нет ли уже документации по этому сценарию

## Во время работы

- не смешивать рефакторинг и изменение поведения без необходимости
- держать изменения минимальными
- не добавлять новые зависимости без явной причины
- не оставлять фиктивные шаги в CI и документации

## Перед завершением задачи

- запустить целевые тесты
- при заметных изменениях запустить весь набор:

```bash
python -m unittest discover -s tests
```

- проверить рабочее дерево:

```bash
git status --short
```

- если менялась логика модулей, обновить `README.md` или `docs/`

## Если меняется стратегия

- проверить конфигурацию стратегии
- проверить расчет сигналов
- проверить влияние на бэктест
- добавить или обновить unit-тест

## Если меняется intraday-логика

- проверить `intraday/engine.py`
- проверить `intraday/smart_scanner.py`
- прогнать `tests.test_intraday`

## Если меняется backtest

- проверить возврат `trades` и `metrics`
- убедиться, что не сломаны ожидаемые ключи метрик
- прогнать `tests.test_engine` и связанные тесты

## Минимальный стандарт готовности

- код читается без дополнительных объяснений
- тесты проходят
- CI-конфигурация не противоречит реальному способу запуска проекта
- документация не вводит в заблуждение
