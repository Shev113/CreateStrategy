# CreateStrategy

Система для анализа и тестирования торговых стратегий на данных MOEX: дневной и интрадей-бэктест, оптимизация, сканирование тикеров, дневник сделок и набор модулей для исследовательской работы.

## Что умеет проект

- загружать и кэшировать исторические данные MOEX
- запускать бэктест стратегий с параметрами риска
- подбирать параметры через оптимизацию
- сканировать тикеры и сектора на торговые сигналы
- выполнять интрадей-анализ и smart scanner для H1
- вести дневник сделок и экспортировать результаты
- отслеживать ожидающие ордера с автоматическим переводом в дневник
- проверяться автоматическими unit-тестами

## Структура проекта

- `core/` - кэш, хранение сессий, базовые вспомогательные сервисы
- `strategy/` - библиотека торговых стратегий и их конфигурация
- `backtest/` - движок бэктеста и расчет метрик
- `intraday/` - интрадей-движок, smart scanner, H1-метрики
- `optimization/` - подбор параметров и связанные сценарии оптимизации
- `screening/` - сканеры тикеров и секторов
- `watchlist/` - логика списков наблюдения
- `diary/` - дневник сделок и post-trade review
- `pending/` - ожидающие ордера с реал-тайм мониторингом и автопереводом в дневник
- `analytics/` - аналитические отчеты и визуализация
- `signals/` - построение и обработка сигналов
- `news/`, `market/`, `monitoring/`, `automation/`, `cloud/` - вспомогательные прикладные модули
- `tests/` - unit-тесты по стратегиям, движкам и вспомогательной логике
- `.github/workflows/main.yml` - CI workflow для GitHub Actions

## Запуск приложения

Основная точка входа находится в корне проекта: это GUI-скрипт, в котором создается `Tk()` и есть блок `if __name__ == "__main__"`.

Для локального запуска используйте корневой файл приложения:

```bash
python CreateStrategyTrading.py
```

Если не уверены, какой файл является точкой входа, ищите корневой Python-файл с инициализацией окна `Tk()`.

Если запускаете не из корня проекта, убедитесь, что рабочая директория содержит все пакеты проекта и `requirements.txt`.

## Запуск тестов

В проекте используется `unittest`.

Запуск всего тестового набора:

```bash
python -m unittest discover -s tests
```

Запуск одного модуля:

```bash
python -m unittest tests.test_intraday -v
```

На текущем этапе локальный набор проходит полностью:

- `260` тестов
- `59` skipped
- статус `OK`

Настроен GitHub Actions workflow:

- файл: `.github/workflows/main.yml`

## Основные сценарии использования

### Анализ и бэктест

- загрузка исторических свечей по тикеру
- запуск стратегии с заданными параметрами
- расчет метрик и визуализация результата

### Оптимизация

- подбор параметров стратегии
- просмотр лучших комбинаций
- применение найденных настроек к бэктесту

### Скринер

- поиск торговых сигналов по списку тикеров
- фильтрация по стратегиям
- экспорт результатов

### Интрадей

- H1-сканирование
- smart scanner для продвинутых стратегий
- композитная оценка результатов по intraday-метрикам

### Дневник сделок

- хранение сделок
- контроль стопов и тейков
- экспорт и импорт данных

### Ожидание сделки

- лимитные ордера на вход по уровню сигнала
- фоновый мониторинг каждые 5 минут
- автоматический перевод сработавшего ордера в дневник
- уведомления при активации ордера
- хранение в `results/pending_trades.json`
- синхронизация с Яндекс.Диском

## Параметры стратегий

Наиболее часто используемые параметры:

- `capital` - стартовый капитал
- `risk_per_trade` - риск на сделку
- `atr_sl` - множитель ATR для стоп-лосса
- `atr_tp` - множитель ATR для тейк-профита
- `min_hits` - минимальное число касаний уровня
- `max_hold` - максимальная длительность удержания позиции
- `commission` - комиссия
- `entry_type` - способ входа в сделку

Дополнительные группы параметров:

- трейлинг-стоп
- частичное закрытие позиции
- pivot-уровни
- ансамбли стратегий
- multi-timeframe фильтры
- управление размером позиции

## Правила разработки

- перед изменениями в торговой логике запускать `python -m unittest discover -s tests`
- изменения в `strategy/`, `backtest/` и `intraday/` проверять минимум целевым модулем тестов
- новые зависимости добавлять в `requirements.txt`
- изменения, влияющие на поведение движка, отражать в README и тестах
- CI должен оставаться зеленым после каждого логически завершенного изменения
